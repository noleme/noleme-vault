# JSR-330 DI Support

Noleme Vault provides support for JSR-330 (Dependency Injection for Java) standard annotations. This allows you to use a more programmatic and annotation-driven approach to dependency injection alongside or instead of YAML configuration.

## Supported Annotations

Vault supports the following annotations from `jakarta.inject` (and also `javax.inject` for backward compatibility):

* `@Inject`: Used on constructors or fields to identify injectable dependencies.
* `@Named`: Used to qualify dependencies by name.
* `@Singleton`: Used to mark a service as a singleton.
* `@Qualifier`: Used to create custom qualifier annotations.

## Constructor Injection

You can use `@Inject` on a constructor to have Vault automatically resolve and inject its parameters.

```java
public class MyService {
    private final MyRepository repository;

    @Inject
    public MyService(MyRepository repository) {
        this.repository = repository;
    }
}
```

If a class has multiple constructors, only one can be marked with `@Inject`. If no constructor is marked, Vault will attempt to use a no-arg constructor.

## Field Injection

Vault also supports field injection.

```java
public class MyService {
    @Inject
    private MyRepository repository;
}
```

Fields are injected immediately after the instance is created.

## Qualifier Support (@Named)

When multiple services of the same type are registered, you can use `@Named` to specify which one should be injected.

```java
public class MyService {
    @Inject
    public MyService(@Named("main_repository") MyRepository repository) {
        // ...
    }
}
```

## Singleton Scope

You can use `@Singleton` to ensure that only one instance of a class is created by the Vault.

```java
@Singleton
public class MySingletonService {
    // ...
}
```

## Generic Modules (@Provides)

While not strictly part of JSR-330, Vault supports a module pattern similar to other DI frameworks using a custom `@Provides` annotation.

```java
public class MyModule {
    @Provides
    @Singleton
    public MyRepository provideRepository() {
        return new MyRepository("connection-string");
    }

    @Provides
    public MyService provideService(MyRepository repository) {
        return new MyService(repository);
    }
}
```

To use a module, pass an instance of it to the `VaultBuilder`:

```java
var vault = Vault.builder()
    .with(new MyModule())
    .build();
```

This enables using `noleme-vault` in a similar fashion to Google Guice, define your provider methods in module classes, then register all Modules, and compile the graph.

## Manual Injection

If you have an existing object that was not created by Vault, you can still perform field injection on it using the `inject` method:

```java
MyService service = new MyService();
vault.inject(service); // This will inject all @Inject-annotated fields
```

## Integration with YAML

JSR-330 based services and YAML-defined services can be used together seamlessly. A service defined in YAML can be injected into a JSR-330 service, and vice versa.

When using `Vault.instance(MyService.class)`, Vault will attempt to find a registration for `MyService`. If it's not found in the registered services (from YAML or modules), it will attempt to instantiate it using its `@Inject` or no-arg constructor, effectively treating it as a just-in-time (JIT) binding.
